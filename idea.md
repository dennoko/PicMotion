# ツール構想: 画像からのHumanoidポーズ生成拡張 (PicMotion - 仮称)

## 1. ツール概要 (Overview)
- **目的**: 1枚の人物写真（画像）から写っている人物のポーズを推定し、UnityのHumanoid互換アニメーションクリップ（.anim）としてワンクリックで書き出すこと。
- **ターゲット**: VRChat向けのアバター改変ユーザー、アニメーション制作においてポーズのベースを素早く作成したいクリエイター。
- **形態**: Unityエディタ拡張（Editor Window機能として実装）。外部サーバーや別のソフトウェアを介さず、Unity内で完結する完全オフライン設計。

---

## 2. コア要件 (Requirements)

### 2.1 機能要件 (Must-have)
1. **画像読み込み機能**
   - エディタウィンドウ上で対象となる画像を選択（ドラッグ＆ドロップ、またはファイルダイアログ）し、TEXTUREとして読み込めること。
   - `PNG`, `JPG` などの標準的な画像形式をサポート。
2. **ポーズ推定推論機能 (AI Inference)**
   - 読み込んだ画像をAIモデルに入力し、人体の各関節（ランドマーク）の2D位置座標(X, Y)を取得できること。
   - Unityエディタがフリーズしないよう、推論処理は非同期（またはEditorCoroutine等）で行い、プログレスバーを表示する。
3. **座標 → 回転変換ロジック (Kinematics)**
   - 取得した各関節の2D位置情報から、関節同士のつながり（ボーン）の方向ベクトルを計算する。Z軸（深度）は初期フェーズではヒューリスティック推定（ボーン長比率による幾何学的推定）で補完する。
   - ユーザーが指定した**HumanoidアバターのPrefab**からTポーズ（デフォルトボーン回転）を取得し、それを基準として方向ベクトルからローカル回転角度（Quaternion）を逆算する。
4. **プレビュー機能**
   - 推論結果のボーン構造（スケルトン）をSceneビュー、またはカスタムプレビュー領域に線（Gizmos / Handles）として簡略描画し、生成前に推定結果を確認できること。
5. **アニメーション保存機能**
   - 計算された各ボーンの回転情報を1フレームのキーフレーム（AnimationCurve）としてまとめ、AnimationClipアセットとして指定フォルダに保存できること。

### 2.2 非機能要件 (Nice-to-have / Constraints)
1. **完全オフライン動作**: 外部API（クラウドサービスなど）に画像を送信せず、ローカル環境のUnity内で推論が完結すること（セキュリティと手軽さの担保）。
2. **直感的なUI/UX**: ユーザーが複雑な設定を必要とせず、「画像を選ぶ」→「アバター指定」→「推論」→「保存」の最短ステップで`.anim`が生成されること。
3. **ハイパフォーマンス**: 推論モデルのロードと実行が数秒以内に完了すること。GPU ComputeShaderバックエンドを優先し、利用不可時はCPUバックエンドへ自動フォールバックする。
4. **エッジケース耐性**: 以下の異常入力に対して適切なエラーメッセージを表示し、クラッシュせずに処理を中断する。
   - 人物が写っていない画像
   - 複数人が写っている画像（最も信頼度の高い1人を自動選択、または警告表示）
   - 関節の一部が遮蔽されている場合（Visibility閾値でスキップまたはデフォルト値で補完）

## 3. アプリケーションアーキテクチャ (Architecture & Design Principles)

プロジェクトは**SOLID原則**に基づき、保守性と拡張性（例：将来的な別AIモデルの導入・変更のしやすさ）を重視したモジュール構造とします。各機能ドメインを疎結合にし、インターフェースを通じて連携させます。

### 3.1 モジュール分割と責任の所在 (SRP: 単一責任の原則)
- `Core`: ドメインオブジェクト（ランドマーク座標データ、ボーン回転データなどの共通構造体）を定義。
- `PreProcessing`: 画像(.Texture2D)からの入力データ生成処理（リサイズ・正規化・Tensor化）。
- `Inference`: AI推論エンジンのラップ処理。前処理済み入力からランドマークを出力する。
- `Kinematics`: 最も複雑なドメインロジック。ランドマーク位置情報から、ベクトル計算・IKを駆使してHumanoid用ローカル回転(Quaternion)に解決する処理。
- `Export`: 計算後の姿勢データを `.anim` ファイル等にシリアライズ・保存する処理。
- `UI`: エディタウィンドウ描画と、ユーザーアクションをトリガーにした上記のパイプラインの指揮（コントローラー）。

### 3.2 インターフェース依存と拡張性 (DIP: 依存性逆転 / OCP: 開放閉鎖の原則)
パイプラインの中心は具象クラスではなく、以下のようなインターフェース群で結合します。
- `IImagePreprocessor`: 画像変換モジュールを抽象化。
- `IPoseEstimator`: 推論エンジンを抽象化。Sentis実装だけでなく、ONNXRuntimeなど他エンジンへも入れ替えが容易になる。
- `IKinematicsSolver`: 座標計算アルゴリズムを抽象化。より精度の高い方式に変更する際も影響を局所化できる。
- `IAnimationExporter`: 保存ロジックを抽象化。

## 4. 技術スタック (Technology Stack)

Unityのエディタ拡張として実装するために、以下の技術・ライブラリを組み合わせる。

### 4.1 AI推論エンジン: Unity Barracuda (`com.unity.barracuda`)
- **使用理由**: Unity公式の軽量ニューラルネットワーク推論ライブラリ。VRChat推奨バージョン（Unity 2022.3.22f1）で動作可能。後継のSentisは同バージョンでは利用不可のため、Barracudaを採用。
- **バックエンド戦略**: GPU (`ComputePrecompiled`) を優先使用し、利用不可時はCPU (`CSharpBurst`) へ自動フォールバック。

### 4.2 姿勢推定AIモデル: RTMPose-Wholebody (ONNX版)
- **使用理由**: OpenMMLab (mmpose) が開発した最新世代のトップダウン型ポーズ推定モデル。COCO benchmarkで最高精度クラスを達成しており、関節位置のブレが少ない。
- **関節数**: Whole-Body版は**133点**（身体・顔・手を含む）。体幹のキーポイントが豊富で、Hips等の計算精度が向上する。
- **ONNX対応**: `mmdeploy` 経由でONNX形式に公式エクスポート可能。Unity Barracudaとの互換性が高い。
- **Z軸(深度)の扱い**: RTMPoseは2D推定モデルのため、Z軸は含まれない。以下の段階的戦略で対処する：
  - **Phase 1 (初期)**: 腕や脚の見かけの長さ比率から幾何学的にZ値をヒューリスティック推定する（例：肘が肩より手前にいるかを上腕の見かけ長から判別）。
  - **Phase 2 (将来)**: 深度推定モデル（MiDaS等）を追加パイプラインとして導入し、ピクセル単位の深度マップからZ情報を補完する。

### 4.3 座標変換・IK処理: Unity C# / 独自数学ロジック
- **基準姿勢の取得**: ユーザーが指定したHumanoidアバターの**Prefab参照**から、`Avatar`アセットのデフォルトボーン回転を直接取得する。
- **座標系変換ルール**: AIモデルの出力座標系（画像座標: 左上原点, Y↓）→ Unityの座標系（Y↑, Z奥行）への変換: `(model_x → unity_x, model_y → -unity_y, heuristic_z → -unity_z)`
- **計算アプローチ**:
  - 推論された133のランドマークのうち、Unity Humanoidに必要な部位（Hips, Spine, Chest, Neck, Head, Shoulders, Arms, Legs等）をマッピング。
  - 親ボーンから子ボーンへの方向ベクトル（例：肩Aから肘Bへのベクトル `AB`）を算出。
  - `Quaternion.FromToRotation` や `Quaternion.LookRotation` を用い、アバターの初期姿勢ベクトルからの差分回転（LocalRotation）を計算する。ひねり（Roll）の解決には補助的なベクトル（ヒンジ軸の計算）が必要。

### 4.4 アセット生成・保存API群 (UnityEditor)
- `EditorWindow`: 独自のUIパネル（モデル設定、画像ドロップエリア、書き出しボタン）の作成。
- `HumanPoseHandler`: ボーン回転をスケルトンに適用後、Humanoidマッスル値を読み取る（ボーン回転→マッスル値変換）。
- `AnimationClip` + `EditorCurveBinding` + `AnimationUtility.SetEditorCurve`: マッスル値を`Animator`型のHumanoidプロパティとしてカーブに書き込み。
- `AssetDatabase.CreateAsset`: クリップのファイル（`.anim`）としての永続化。

## 5. システムの処理フロー詳細 (Pipeline)

前述のアーキテクチャに基づく一連のパイプラインです。

1. **[Input] データ準備** (`PicMotionWindow`)
   - ユーザーからの画像入力、HumanoidアバターPrefabの参照セット。
2. **[Pre-process] 画像の前処理** (`IImagePreprocessor`)
   - 画像をRTMPoseが要求する入力解像度(192x256)にRenderTexture経由でリサイズし、ImageNet mean/std正規化を適用したNHWC形式のfloat配列を生成。
3. **[Inference] 推論実行** (`IPoseEstimator`)
   - Unity Barracuda の IWorker を使用してRTMPose-Wholebody推論を実行。133カ所の関節の (x, y, score) 情報を抽出 (`PoseLandmark[]`)。
4. **[Z-Estimation] 深度推定** (`IDepthEstimator`)
   - 肩幅を基準長とし、各ボーンの見かけの2D長と想定3D長を比較。直角三角形の原理からZ値をヒューリスティック推定し、擬似3D座標を構築。
5. **[Post-process] キネマティクス計算** (`IKinematicsSolver`)
   - アバターPrefabを一時インスタンス化し、AnimatorからTポーズのデフォルトボーン回転を取得。
   - ランドマークから仮想ジョイント(Hips/Spine/Chest/Neck)を補間生成し、各関節の方向ベクトルを算出。
   - `Quaternion.FromToRotation` でTポーズ方向からターゲット方向への差分回転を計算。
6. **[Export] アニメーション書き出し** (`IAnimationExporter`)
   - ボーン回転をスケルトンに適用後、`HumanPoseHandler`でHumanoidマッスル値を読み取り、各マッスル名の`EditorCurveBinding`としてAnimationClipに書き込み、AssetDatabase経由で保存。

## 6. 課題と将来の拡張アイデア (Future Perspectives)

- **Z軸精度の向上**: 初期のヒューリスティックZ推定から、MiDaS等の深度推定モデルを組み合わせた高精度3D復元への段階的移行。
- **IK補正**: 両足の接地を安定させるための簡単なIK補正（足の高さ揃え）や、手動での微調整スライダーをエディタ上に設ける。
- **動画対応**: 1枚の画像だけでなく、動画ファイル（.mp4など）からフレーム単位の連続推論を行い、フルアニメーションの作成を可能にする。
- **手指のポーズ**: RTMPose-Wholebody は133点中42点が手のランドマークであるため、将来的にはこの情報を活用して指先のポーズも反映できるようにする。
